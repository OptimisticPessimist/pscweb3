# 実装計画: 台本アップロード時の422エラーおよびリビジョン不整合の修正

## 1. 目的
- ユーザーからの報告にある、台本アップロード時の500エラーや422エラーを完全に解消すること。
- システムの台本リビジョン（Rev.32）とDiscord通知のリビジョン（Rev.25）の乖離原因を特定し、将来的な通知のサイレント失敗を防ぐためのロジックを修正すること。

## 2. 原因分析
1. **422 Validation Error**:
   前回の改修で、バックエンドAPI（`/api/scripts/{project_id}/upload`）の必須パラメータ名を `file` からフロントエンド側の主流だった `script_file` に変更しました。しかし、別画面にあたる `ScriptUploadPage.tsx` では依然として `file` というキー名でデータを送信していたため、バックエンド側でファイルの受け取りに失敗し（Validation Error = 422）アップロードできませんでした。
2. **リビジョンの不整合 (Rev.25 -> Rev.32)**:
   API経由での台本パースとデータベースへの保存（=リビジョンの増加）自体は正常に行われていました。しかし、DBへのコミット完了「後」に呼び出されるバックグラウンド処理（`send_script_notification`）の中で、生成されたPDFのサイズがDiscordのWebhook容量上限（通常8MB）を超過したか、一時的な通信エラーが発生したことにより、通知そのものが「サイレントエラー」として失敗していました。結果、DB内部だけリビジョンが進行してしまいました（※これらは500エラー修正前、旧仕様での複数回アップロード施行時を含む）。

## 3. 解決策
1. フロントエンドからのファイルアップロード時にキー名が `file` でも `script_file` でも受け取れるように、または一貫して正しいキーを送るように修正します。
2. Discordへのファイル送信（PDF等）が失敗した場合、その詳細な内容をサーバーログやDiscordのメッセージ本文（ファイル添付なし）としてでも明示的に出力・通知できるように連携処理を強化します。

## 4. 修正対象ファイル

### `frontend/src/features/scripts/pages/ScriptUploadPage.tsx`
- **修正内容**:
  アップロードのフォームデータ作成処理において、`formData.append('file', file);` としている箇所を、`formData.append('script_file', file);` に変更します。

### `backend/src/api/scripts.py`
- **修正内容**:
  `script_file: UploadFile = File(...)` に対して `alias="file"` を付与し、古いキャッシュや別クライアントから `file` という名前でリクエストが来ても後方互換性で適切に処理できるようにします。

### `backend/src/services/discord.py` または関連サービス
- **修正内容**:
  Discord通知の例外処理で、単にログを吐いて終わるのではなく、もし `413 Payload Too Large` 等で失敗した場合は、スタックトレースや具体的なエラー内容（"PDF File too large for Discord"など）をログにより詳細に出力し、原因箇所を特定しやすくします（ユーザーへの画面表示は、すでにDB保存が完了しているため通常通りで問題ありません）。

## 5. テスト・検証手順
1. `ScriptUploadPage.tsx` から台本（.fountain）をアップロードし、422エラーにならず正常終了するか確認します。
2. これまでに作成したユニットテスト群（`test_api_scripts.py`）が通ることを確認します。
3. 問題がなければコミットを行います。
