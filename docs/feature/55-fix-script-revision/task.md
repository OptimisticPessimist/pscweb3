# タスク: 台本アップロード時の422エラーおよびリビジョン不整合の修正

## 背景と課題
ユーザーより以下の問題報告がありました。
1. `Upload failed: Request failed with status code 422` のエラーが発生する。
2. Discordに送信されている台本の最新版がRev.25なのに対し、システム上のリビジョンはRev.32に飛んでしまっている。

調査の結果、それぞれの原因は以下の通りであることが判明しました。
- **422エラーの原因**: 前回の改修でバックエンドAPI（`/api/scripts/{project_id}/upload`）のファイル受け取りパラメータ名を `file` から `script_file` に変更しましたが、フロントエンドの別画面である `ScriptUploadPage.tsx` が古い `file` というキー名のままデータ送信を行っており、バックエンドのバリデーションでは「必須パラメータ `script_file` がない」と判定されて弾かれていました。
- **リビジョン不整合（Rev.25 -> 32）の原因**: 台本ファイル自体は問題なくパースされ、データベースのトランザクションはコミットされてRev.32までシステム内部では更新成功していました。しかし、その後のバックグラウンドでの「Discord通知＆PDF送信処理(`send_script_notification`)」において、PDFのファイルサイズがDiscord側の上限（8MB等）を超過していたか、一時的な通信エラーなどの原因で通知自体がサイレントエラーとして失敗していました。システムは成功しDiscordのみ失敗したため不整合が生じました。

## 作業内容
- [ ] フロントエンドの `ScriptUploadPage.tsx` を修正し、ファイル送信時のパラメータ名を `file` から `script_file` に統一する。
- [ ] バックエンドの `src/api/scripts.py` において、`script_file` パラメータに `alias="file"` を付与し、古いクライアントからのリクエストでも422エラーにならずに安全に受け取れるよう後方互換性を持たせる。
- [ ] DiscordへのPDF通知失敗の根本原因の特定とログ改善（PDF転送時のエラーログをさらに明確に残し、不具合の追求を容易にする）。
- [ ] フロントエンド側の動作確認。
- [ ] コミット作成。
