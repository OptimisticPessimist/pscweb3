# 実装計画

## 目標
日程調整画面において、すべてのプロジェクトメンバー（`viewer`権限や後から参加したメンバーを含む）が自分自身のスケジュールを入力できるように修正する。

### 仕様変更の合理性について
現在、システムロールとして `owner`, `editor`, `viewer` が存在していますが、実際の劇団やプロジェクト運用においては以下の特性があります。
1. **人員の必要性の管理**: 誰が稽古に必要かはシステム上の権限（編集権限の有無）ではなく、「役職（演出、音響など）」や「配役（キャストとしての割り当て）」で管理されるのが実態です。
2. **`viewer` の役割**: プロジェクトの設定変更や台本のアップロード等の「管理・編集作業」はさせたくないが、連絡事項を受け取り、自分の出欠だけは入力してほしいメンバー（例: 客演の役者、当日のみの手伝いスタッフなど）を想定した場合、現在の「`viewer` は出欠や日程調整を入力できない」という仕様は不便を強いることになります。

したがって、**「プロジェクトの管理作業の権限（owner/editor/viewer）」と「個人のスケジュール提出の権限（全員可能）」は切り離して機能を提供するのが運用上合理的**であると判断しました。

## 現状の課題
1. **バックエンド**: `answer_poll` エンドポイントで `viewer` 権限からの回答を `403 Forbidden` としてブロックしている。
2. **フロントエンド**: `SchedulePollDetailPage.tsx` で、回答入力列（「あなたの回答」など）や入力操作が `isEditorOrOwner` の場合にのみ表示・許可されている。

## 修正内容

### 1. バックエンド (`schedule_polls.py`)
対象ファイル： `backend/src/api/schedule_polls.py`
- `answer_poll` メソッド内の権限チェックで、`viewer` 権限を弾く処理を削除する。
- メンバーとしてプロジェクトに参加していれば誰でも回答できるようにする。

### 2. フロントエンド (`SchedulePollDetailPage.tsx`)
対象ファイル： `frontend/src/features/schedule_polls/pages/SchedulePollDetailPage.tsx`
- 新しく `isMember` という判定用プロパティを追加し、現在のユーザーがプロジェクトメンバーであれば `true` にする。
- 回答用のテーブルのヘッダーおよびセルの表示条件を `isEditorOrOwner` から `isMember` に変更する。
- 削除ボタンやリマインド停止機能など、管理者向け機能については引き続き `isEditorOrOwner` を適用する。

## 確認方法（手動テスト）
1. `viewer` 権限のユーザーで日程調整画面にアクセスする。
2. そのユーザーで「あなたの回答」列が表示され、回答（〇、△、×）を入力できることを確認する。
3. そのユーザーでは「削除」や「自動リマインド停止」等の管理者機能が表示されないことを確認する。
